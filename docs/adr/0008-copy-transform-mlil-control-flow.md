# Use copy transformations for MLIL control-flow rewrites

DispatchThis will implement MLIL rewrites that change control-flow edges by copying the current function with Binary Ninja's copy-transformation APIs, resolving destinations through copied source-block labels, and installing the replacement through `AnalysisContext.set_mlil_function()`. Branch condition translation is the first use and deflattening is the next; expression-only rewrites may continue to use `replace_expr`. Each transformation using this backend is atomic: if any selected control-flow rewrite fails or the replacement cannot be installed, DispatchThis discards it instead of publishing a partial graph or replacement-dependent receipt. Replacement-dependent branch cleanup receipts are confirmed only after successful installation, while a no-change scan of the current MLIL may confirm cleanup without a replacement commit; deflatten metadata and stability will adopt the same rule when deflattening moves to this backend. Binary Ninja releases before 5.3 and older assignment fallbacks are out of scope. Validation compares recovered CFG edges and target blocks; HLIL is checked for readable output but is not treated as a stable textual interface. This preserves Binary Ninja's label and IL-mapping semantics without changing resolver plans, workflow phase ordering, or cleanup ownership.
